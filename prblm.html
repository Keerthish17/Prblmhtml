The Root Problem

Why You Get Stuck at O(n²)

Every time you see a problem, your brain reads the question and immediately builds a loop around it. The question becomes the outer loop. Then you scan everything inside.

❌ Default Brain
For every rotation:
    check all elements

For every query:
    update all indices

For every subarray:
    compute the answer

For every operation:
    touch every position in range
✓ Rewired Brain
For every element:
    find all rotations it affects

For every update:
    record where it starts & stops

For every operation:
    mark the interval, not the points

Let prefix sums spread
the effect automatically.

The Single Root Insight

In almost every O(n²) problem that can be reduced, there is one overlooked fact:

One element or operation does not affect one point — it affects a continuous interval of answers.

Once you see the interval, you stop simulating it point-by-point. You record the start and stop of the influence. A single prefix-sum pass reconstructs everything.

This is the only mental shift. Everything else is mechanics.

The Framework

How to Approach Any Problem

Every time you see a new problem, run this checklist in your head before coding anything.

🧠 THE 6-QUESTION RECOGNITION CHECKLIST

1

What is contributing? — An operation? An element? A trip? A meeting?

2

What receives the contribution? — Indices? Time slots? Positions? Queries?

3

When does contribution start? — At which index/time/position?

4

When does contribution stop? — At which index/time/position?

5

Is the contribution uniform and continuous? — Same value across the whole range?

6

Can it be represented as an interval [l, r]? — If YES → Difference Array + Prefix Sum.

The 3-Step Execution Template

STEP 1: Build diff array of size (n+1), all zeros.

STEP 2: For each operation (l, r, val):
    diff[l]   += val     ← START: influence begins here
    diff[r+1] -= val     ← STOP:  influence ends after r

STEP 3: Prefix sum sweep:
    running = 0
    for i in 0..n:
        running += diff[i]
        answer[i] = running      ← running IS the answer at i
Core AHA

"diff[l] += val" is not a formula. It means: starting from l, the world gets val more.
"diff[r+1] -= val" means: starting from r+1, that extra val is gone.
The prefix sum just carries the running total forward. Overlaps are free — they stack automatically.

Range Addition

Add val to every index from l to r, k times. Return final array.

Difference Array Prefix Sum

Problem Statement

Input: n = 5, operations = [[1,3,2], [0,2,3], [2,4,1]]
Each operation [l, r, val] means: add val to every index from l to r.
Output: Final array after all operations.

Brute Force Thinking

You read "add val to every index from l to r" — so you loop l to r and add val each time.

array = [0, 0, 0, 0, 0]
for op in operations:
    for i in range(op.l, op.r + 1):
        array[i] += op.val
# O(n × k) — bad when n=10^5, k=10^5
❌ Old View
For every operation,
  touch every index
  in [l, r].

Cost: O(range size)
per operation.
✓ New View
For every operation,
  record START at l,
  record STOP at r+1.

Cost: O(1)
per operation.
AHA Moment

"This operation's effect is continuous across [l,r]. I don't need to touch every point — I only need to record where the influence begins and where it ends."

Each booking: [first, last, seats] — seats reserved on flights first through last (1-indexed).

AHA Moment

"Each booking doesn't just affect one flight — it occupies a continuous range of flights. The booking is an interval. I record its start and its stop. I don't touch every flight in between."

Minimum Number of K-Operations to Reduce Array

You can subtract 1 from any subarray of length k. Find minimum ops to make all elements ≤ 0.

Difference Array Greedy + Operations

Problem Statement

Input: arr = [3, 1, 2, 4, 1], k = 3
Each operation: choose any subarray of length exactly k, subtract 1 from every element.
Output: Minimum operations to make every element ≤ 0. Return -1 if impossible.

Why This Is A Contribution Problem

When you perform an operation starting at index i, it affects indices i, i+1, ..., i+k-1. Each operation is a range update of -1 over a window of size k.

The key insight: process greedily left to right. At each index, apply exactly as many operations as needed to bring it to ≤ 0. Record the effect of those operations on future indices using the diff array.

❌ Old View
For each operation:
  pick a subarray of length k
  subtract 1 from all k elements
  
Simulate each subtraction.
O(n × ops)
✓ New View
At index i, I need 'x' more ops.
Apply x ops starting at i.
They affect i to i+k-1.

diff[i]   -= x  (record start)
diff[i+k] += x  (record stop)
Running total = ops still active.
AHA Moment

"An operation starting at i affects a continuous window [i, i+k-1]. Instead of simulating each subtraction, I record the operation's range using the diff array. The running total tells me how many operations are currently reducing each element."

Full Dry Run

arr = [3, 1, 2, 4, 1], k = 3

diff array (size n+1 = 6), tracks ops currently active:

diff    = [0, 0, 0, 0, 0, 0]
running = 0   (ops currently active)

Check: index 4 is the last valid start for k=3 (needs indices 4,5,6 but array is size 5). If we still need ops at index 3, it means i+k-1 = 3+2 = 5 which is out of bounds → return -1 if impossible.

Meeting Rooms II — Minimum Rooms

Meetings have start and end times. Find minimum conference rooms needed simultaneously.

Sweep Line Events

Problem Statement

Input: intervals = [[0,30],[5,10],[15,20]]
Output: Minimum number of conference rooms required (= max simultaneous meetings).

Key Realization

Each meeting occupies a room for a continuous interval of time. The number of rooms needed at any moment = number of overlapping intervals at that moment. This is exactly the prefix sum of +1/-1 events.

❌ Old View
For every pair of meetings:
  check if they overlap
  
Track groups of overlapping.
O(n²)
✓ New View
Each meeting = interval.
Start = +1 room needed.
End   = -1 room freed.

Sort events by time.
Max running total = answer.
AHA Moment

"I don't need to compare meeting pairs. Each meeting contributes +1 to a continuous time interval. The peak of the running sum IS the minimum rooms needed. Overlap detection becomes a prefix max problem."

Full Dry Run — Event Processing

Meetings: [0,30], [5,10], [15,20]

Create events:

Events (sorted by time):
(time=0,  type=START, +1)   ← meeting [0,30] begins
(time=5,  type=START, +1)   ← meeting [5,10] begins
(time=10, type=END,   -1)   ← meeting [5,10] ends
(time=15, type=START, +1)   ← meeting [15,20] begins
(time=20, type=END,   -1)   ← meeting [15,20] ends
(time=30, type=END,   -1)   ← meeting [0,30] ends

Note: when times tie, process END before START
(freeing a room before allocating one)

Maximize Sum After K Negations

Apply exactly k operations. Each op negates a subarray of length 1. Maximize total sum.

K Operations Contribution

Problem Statement (Generalized)

You have array arr. You must perform exactly k operations. Each operation: pick a contiguous subarray of length L, multiply all elements by -1.
Maximize the total sum after all k operations.

The Contribution Thinking Here

Each element can be negated an even or odd number of times. If negated odd times → sign flips. If even times → back to original. The total number of times an element is negated = number of operations whose window covers it.

This is a range coverage counting problem. How many of the k operation windows cover index i?

AHA Moment

"An operation doesn't just touch one element — it covers a range. The parity of coverage (odd/even times covered) determines the final sign of each element. I want to arrange k windows to maximize sum, which means: maximally cover negative elements and minimally (or evenly) cover positive ones."

Simpler version dry run: arr = [-4, 2, -3, 1], k=2, L=2 (negate any window of length 2)

All possible windows: [0,1], [1,2], [2,3]

What contributes to index i? Any window starting at position max(0, i-L+1) to min(i, n-L).

ElementValueWindows covering itSign if negated odd times
0-4[0,1]+4 (want odd coverage!)
1+2[0,1],[1,2]-2 (want even coverage)
2-3[1,2],[2,3]+3 (want odd coverage!)
3+1[2,3]-1 (want even coverage)

Greedy: use window [0,1] and [2,3]. Negations: [-4→+4, 2→-2, -3→+3, 1→-1]. Sum = 4-2+3-1 = 4.

Without operations: -4+2-3+1 = -4. Huge gain from contribution thinking!

Minimum Value After Removal Operations

Remove elements using k operations. Each op removes one occurrence. Find subarray with min sum after all removals.

K Operations Prefix Sum Range Contribution

Problem Statement

Input: arr = [1, 3, 2, 5, 4], k = 2
Remove exactly k elements (set to 0). Then find the subarray with the minimum sum.
Output: Minimum possible subarray sum after optimal removals.

The Insight: Which Removal Affects Which Subarrays?

Removing element at index j reduces the sum of every subarray that includes index j. Which subarrays include j? Those starting at 0..j and ending at j..n-1. That's a 2D contribution problem — but we simplify by asking: which single element's removal helps the most?

Contribution of removing element at index j = its value (because subarrays containing j decrease by arr[j]).

Greedy: Remove the k largest positive elements (they hurt minimum subarrays the most when present). Then find min subarray of the resulting array.

AHA Moment

"Removing element j affects ALL subarrays containing j — a continuous range of subarrays. The contribution of removal = the element's value. Removing the largest elements maximally shifts the minimum subarray sum downward."

Full Dry Run

arr = [1, 3, 2, 5, 4], k = 2

Step 1: Which 2 elements to remove to minimize the best subarray?
Remove largest positives: 5 (index 3) and 4 (index 4).

Modified arr = [1, 3, 2, 0, 0]

Step 2: Find minimum subarray sum (Kadane's variant for minimum):

IndexValueMin ending hereGlobal min

Answer: minimum subarray sum = 1 (just element at index 0).

Without removals: minimum subarray = [1] = 1. The removals helped for longer subarrays but the minimum is already at [1].

Smallest Rotation with Highest Score

Rotate array by k positions. Score = count of indices where arr[i] ≤ i. Find k that maximizes score.

Rotation Range Difference Array Prefix Sum

Problem Statement

Input: arr = [2, 3, 1, 4, 0], n=5
Rotation by k: element at position i moves to position (i - k + n) % n.
Score(k) = number of indices where rotated arr[i] ≤ i.
Output: Smallest k that gives maximum score.

The Hard Part: What Does Rotation Do?

After rotating by k, element originally at index i is now at index (i - k + n) % n. Call new index j = (i - k + n) % n. The element scores a point if arr[i] ≤ j = arr[i] ≤ (i - k + n) % n.

For Which Values of k Does arr[i] Score?

Element arr[i] scores a point for rotation k when: arr[i] ≤ (i - k + n) % n.

This simplifies to: element i scores for k in a specific range. Let v = arr[i]:

  • If v == 0: scores for ALL k in [0, n-1]
  • If v ≤ i: scores for k in [0, i-v] and [i+1, n-1] (wraps around)
  • If v > i: scores for k in [i+1, n-v] (or empty if n-v < i+1)
AHA Moment — The Hardest Shift

OLD VIEW: "For every rotation k, check all n elements to compute score(k)." → O(n²)

NEW VIEW: "For every element, find the range of k values where it contributes +1 to score. Mark that range in a diff array. Sum up to get score(k) for all k at once." → O(n)

Full Dry Run

arr = [2, 3, 1, 4, 0], n = 5

diff array of size n (one slot per rotation 0 to n-1):

diff = [0, 0, 0, 0, 0]
k      0  1  2  3  4
Element i=0, arr[0]=2 (v=2, v > i=0)
Scores for k in [i+1, n-v] = [1, 3]
diff[1] += 1, diff[4] -= 1
diff = [0, 1, 0, 0, -1]
Element i=1, arr[1]=3 (v=3, v > i=1)
Scores for k in [i+1, n-v] = [2, 2]
diff[2] += 1, diff[3] -= 1
diff = [0, 1, 1, -1, -1]
Element i=2, arr[2]=1 (v=1, v ≤ i=2)
Scores for k in [0, i-v]=[0,1] and [i+1, n-1]=[3,4]
diff[0] += 1, diff[2] -= 1, diff[3] += 1, diff[5] -= 1 (out of bounds, ignore)
diff = [1, 1, 0, 0, -1]
Element i=3, arr[3]=4 (v=4, v > i=3)
Scores for k in [i+1, n-v] = [4, 1] — empty (4 > 1), so NO contribution
Element i=4, arr[4]=0 (v=0)
v==0 → scores for ALL k in [0, n-1]
diff[0] += 1, diff[5] -= 1 (out of bounds)
diff = [2, 1, 0, 0, -1]

Final diff = [2, 1, 0, 0, -1]

Complexity: O(n²) brute → O(n) contribution

Maximum Score After K Operations on Subarrays

Apply k operations to choose subarrays and maximize/minimize. The final boss — multi-layer contribution analysis.

K Operations 2D Contribution Difference Array Sweep Line

Problem Statement

Input: arr = [1, 2, 3, 4, 5], k = 3
You may perform exactly k operations. Each operation: choose any subarray, add 1 to every element in it.
Score = sum of the entire array after all k operations.
Output: Maximum possible score. Also: which subarray was chosen in each operation?

Naive vs Contribution

Naive: try all possible (subarray, operation) combinations → exponential.

Insight: Each operation adds 1 to some subarray. To maximize the total sum, each operation should add 1 to the entire array (choose [0, n-1] every time). Total gain = k × n.

But what if operations are constrained — say, each index can receive at most m total additions?

Constrained Version (The Real Challenge)

Each index i can receive at most limits[i] total additions across all k operations. Maximize total sum.

arr = [1,2,3,4,5], limits = [2,1,3,1,2], k = 5

AHA Moment — The Final Boss

OLD VIEW: "For each operation, decide which subarray to pick." → Exponential search.

NEW VIEW: "Each index independently receives some number of additions (0 to limits[i]). What's the maximum additions we can distribute across indices given total budget k? → Greedy: fill indices from highest limit down, never exceed limits[i], total ≤ k. The result is the addition array. Contribution = how many ops cover each index. Represent as diff array."

Full Dry Run

arr=[1,2,3,4,5], limits=[2,1,3,1,2], k=5

Step 1: Find optimal additions per index (greedy — maximize sum means maximize total additions, respecting limits):

additions = min(limits[i], remaining_budget_for_i)

Index 0: limit=2, take 2. k_left = 5-2 = 3
Index 1: limit=1, take 1. k_left = 3-1 = 2
Index 2: limit=3, take 2 (only 2 left). k_left = 2-2 = 0
Index 3: limit=1, can't take (budget=0)
Index 4: limit=2, can't take (budget=0)

additions = [2, 1, 2, 0, 0]

Step 2: Represent additions as operations. We need to express [2,1,2,0,0] as a sum of k=5 subarray operations of +1 each. Use contribution: diff array of additions IS the net effect.

diff of additions = [2-0, 1-2, 2-1, 0-2, 0-0] = [2, -1, 1, -2, 0]

This means: 2 ops start at 0 (but stop at 1 before reaching 2), 1 op starts at 2 (stops at 3). Translating to real operations:

Op1: add 1 to [0,0]
Op2: add 1 to [0,0]
Op3: add 1 to [1,1]
Op4: add 1 to [2,2]
Op5: add 1 to [2,2]

Step 3: Final array = original + additions:

Maximum score = 3+3+5+4+5 = 20 (vs base 15, gained 5 from k=5 ops).

What Made This Level 10?

The contribution runs in two directions: operations contribute to indices (range effect), AND the optimal placement of operations is itself computed by a contribution/greedy analysis of what each index can receive. Double layer. Same pattern, applied twice.

    for i in 0..n:
        running += diff[i]
        answer[i] = f(running)   // running IS the answer
The 5 Trigger Sentences — Burn These In

1. "This element contributes to a whole range." → Difference array.
2. "This update starts here and stops there." → diff[l]+=x, diff[r+1]-=x.
3. "I don't need to apply the update everywhere." → Two writes, not n writes.
4. "I only need to record where influence begins and ends." → Events, not simulation.
5. "The affected region is an interval." → Prefix sum reconstructs everything.
